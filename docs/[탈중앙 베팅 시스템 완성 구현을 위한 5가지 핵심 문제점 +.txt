[탈중앙 베팅 시스템 완성 구현을 위한 5가지 핵심 문제점 + 대응 방안 (T값 포함 상세 예시)]

현재 상황 요약:
- Possible win 값을 통해 예상 체결 평균 배당(expected_avg_odds) 계산 가능
- slippage% 계산 가능
- safeStake(슬리피지 허용 기준 안전 베팅 금액) 계산 로직 구현 중
→ 즉, 1단계(슬리피지 기반 계산 엔진)는 거의 완료 단계

이제 탈중앙 베팅을 “실전에서 안정적으로 운용”하기 위해 반드시 대응해야 할 5가지 핵심 문제를 아래와 같이 정리합니다.


====================================================================
1. 문제 #1: 슬리피지(Slippage) – 표시 배당과 실제 체결 배당 불일치

■ 현상
탈중앙 베팅에서는 표시 배당(displayed odds)과 실제 체결 평균 배당이 다를 수 있음.
베팅 금액이 커질수록 유동성 풀 가격 곡선에 의해 실제 체결 배당이 낮아짐.

■ 구체적 예시
표시 배당: 2.20
베팅 금액(stake): 400
Possible win: 852

expected_avg_odds = 852 / 400 = 2.13
slippage = (2.20 - 2.13) / 2.20 × 100 = 3.18%

→ 표시 배당 기준 기대 수익률보다 실제 수익률이 크게 감소

■ 대응 방안
1) expected_avg_odds = possible_win / stake 계산
2) slippage% 실시간 계산 및 UI 표시
3) T값(슬리피지 허용 기준) 설정
   예: T = 2.5%

4) safeStake 산출 로직
   - stake를 증가시키며 slippage <= T 를 만족하는 최대 금액 탐색
   예:
   stake 100 → slippage 0.4% (OK)
   stake 200 → slippage 0.9% (OK)
   stake 300 → slippage 1.8% (OK)
   stake 400 → slippage 3.18% (초과)

   → safeStake = 300

5) UI 표시:
   - 최대 베팅 금액(maxStake)
   - 추천 최대 베팅 금액(safeStake)
   - 현재 slippage%


====================================================================
2. 문제 #2: 타이밍 리스크 – 계산 시점과 실제 클릭 시점 불일치

■ 현상
safeStake를 계산한 후 사용자가 몇 초 뒤 베팅하면,
그 사이 다른 유저 베팅으로 유동성 풀 상태가 변할 수 있음.

■ 예시
t0 시점 계산:
표시 배당: 2.20
safeStake = 300 (slippage 1.8%)

t0 + 3초:
다른 유저가 먼저 200 베팅
→ 실제 safeStake = 150으로 감소

사용자가 300 그대로 베팅하면
→ 실제 slippage 4% 이상 발생 가능

■ 대응 방안
1) 베팅 버튼 클릭 직전 재Quote 수행
   - possible win 재조회
   - expected_avg_odds 재계산
   - slippage 재검증

2) 재검증 결과 slippage > T 일 경우:
   - 경고 메시지 표시
   - safeStake로 자동 조정 옵션 제공


====================================================================
3. 문제 #3: 유동성 경쟁 리스크 – 다른 유저가 먼저 체결하는 상황

■ 현상
탈중앙 구조는 시장 기반이므로,
동시에 여러 유저가 동일 오즈를 소비할 수 있음.

■ 예시
현재 유동성 기준:
해당 오즈에서 총 가능 유동성 = 500

사용자 safeStake 계산 = 300

하지만 실제:
다른 유저가 먼저 400 체결
→ 남은 유동성 = 100

사용자가 300 넣으면:
일부는 더 낮은 배당으로 체결되거나 부분 체결 발생

■ 대응 방안
1) safeStake 계산 시 버퍼 적용
   예:
   계산된 safeStake = 300
   실제 표시 safeStake = 300 × 0.8 = 240

2) 베팅 직전 남은 유동성 재조회
3) 부족 시 경고 및 stake 자동 감소 안내


====================================================================
4. 문제 #4: 오즈 변동 리스크 – 시장 배당 자체가 변하는 상황

■ 현상
라이브 경기 또는 변동성 높은 시장에서는
슬리피지와 별개로 표시 배당 자체가 변동됨.

■ 예시
t0:
displayed_odds = 2.20
safeStake = 300

t0 + 2초:
displayed_odds = 2.05 로 하락

→ 기존 slippage 계산 및 safeStake 기준이 모두 무효화됨

■ 대응 방안
1) 표시 배당(displayed odds) 실시간 감지
2) 배당 변경 시 자동 재Quote 수행
3) slippage, safeStake 재계산 후 UI 즉시 갱신

4) T값 동적 적용:
   - 프리매치: T = 2.0%
   - 메이저 리그 라이브: T = 3.0%
   - 마이너 리그 라이브: T = 4.0%


====================================================================
5. 문제 #5: 부분 체결 / 미체결 리스크

■ 현상
유동성이 부족하면 일부 금액만 체결되고 나머지는 pending 상태가 될 수 있음.

■ 예시
사용자 stake = 500
즉시 체결 = 320
미체결(pending) = 180

실제 평균 체결 배당은
confirmed_avg_odds = confirmed_win / confirmed_stake
기준으로 재계산 필요

■ 대응 방안
1) 체결 완료 금액 / 미체결 금액 분리 표시
2) 실제 체결 평균 배당 확정 후 재표시
3) 미체결 금액 처리 옵션 제공:
   - 자동 취소
   - 일정 시간 대기 후 취소
   - 더 낮은 배당 허용 여부 선택
